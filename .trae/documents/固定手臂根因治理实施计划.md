# 固定手臂根因治理实施计划

## Summary
- 目标：彻底恢复并固化 fixed-arm 语义，使 `task.arm` 成为任务创建时确定、运行期不可改写的唯一调度真相。
- 本次不是继续加防呆补丁，而是拆除导致 fixed-arm 被破坏的三条根因链：运行时改 arm、按物理现场反推任务 arm、交换成功后再做事后 arm 同步。
- 成功标准：PM2 交换、LLA/LLB 立即补取、Robot 执行守门三条路径都只消费既有 `task.arm`；现场不满足时只等待/回退/告警，不再改写任务 arm，也不再默认换臂跑通。

### 5. fixed-arm 被破坏的根因归纳
- 根因一：`task.arm` 被设计成了既是调度语义，又可被执行现场回写的双重角色。
- 根因二：LL/PM 调度层历史上根据哪只手空闲/哪只手有片反推或重写任务 arm，而不是只校验是否满足既定任务 arm。
- 根因三：PM2 交换路径先执行物理动作，成功后再用 `syncPm2ReturnTaskArm(...)` 做语义补救，说明调度在发起前并没有完成任务语义闭环。
- 根因四：接口和日志层仍保留允许同步实际手指的旧语义，后续维护者容易再次把 fixed-arm 退化为可改字段。

## Proposed Changes

### A. 把 `task.arm` 正式升级为不可变绑定语义
**文件**
- `d:\HLPrj\HL\device\include\UnifiedWaferTask.h`
- `d:\HLPrj\HL\device\include\TaskManager.h`
- `d:\HLPrj\HL\device\src\TaskManager.cpp`

**修改内容**
- 在 `UnifiedWaferTask.h` 中把 `arm` 注释明确为绑定手臂（binding arm），只在任务创建时赋值。
- 在 `TaskManager.h` 中移除或弃用 `updateTaskArm(...)` 的旧语义注释，改成显式禁止运行期改写；若工程允许，优先删除该接口声明；若暂时不能删，则保留为兼容壳并标注 forbidden。
- 在 `TaskManager.cpp` 中保留当前拒绝改写并记录错误的实现，作为过渡期硬防线。

**原因**
- 这是根治 fixed-arm 的总开关；只要运行期还能合法改 `task.arm`，所有上层治理都可能被再次绕开。

### B. 让 PM2 交换在发起前就完成任务语义闭环
**文件**
- `d:\HLPrj\HL\device\src\slot_transfer_cycle_vtm_widget.cpp`

**修改内容**
- 继续以 `tryBuildPm2ExchangePlan(...)` 作为 PM2 交换唯一入口，要求它返回完整任务对：`returnTaskId`、`pendingTaskId`、`getArm`、`putArm`。
- 在 `step 200` 只允许两种交换：`A取/B放` 或 `B取/A放`，且它们必须直接来源于任务对，而不是来源于现场哪只手空/哪只手有片的临时推导。
- 在 `1065/1075` 成功分支中移除对 `syncPm2ReturnTaskArm(...)` 的依赖；交换成功后不再做 arm 同步或二次校对，只依据发起前已锁定的任务对推进后续状态流转。
- 将 `syncPm2ReturnTaskArm(...)` 降级为调试/告警辅助，最终目标是移除该接口，避免它继续以补救工具身份存在。

**原因**
- fixed-arm 的关键不是交换后发现错了别改 arm，而是交换前就不允许语义不闭环的动作发出去。

### C. 统一 LLA/LLB 为按任务 arm 等待，不再按空闲手重排
**文件**
- `d:\HLPrj\HL\device\src\slot_transfer_cycle_vtm_widget.cpp`

**修改内容**
- 保持并补全 `tryPlanImmediateRepick(...)` 的使用范围，让 LLA/LLB 的立即补取都只按 occupiedArm 关联回片任务和同 arm 的下一条 pending task。
- 保持 `isRobotArmOccupiedForLlRequest(...)` 的执行守门角色：目标手臂被占用、已有 in-flight 请求、交换在途时，一律等待或回退，不切换另一只手。
- 复查 LLA/LLB 相关 case，确保不存在遗漏的desiredArm 不可用 -> 改用 otherArm路径。

**原因**
- LL 是 fixed-arm 最容易被现场空闲手诱导破坏的入口，必须从空闲优先改成绑定 arm 优先。

### D. 明确 Robot/TM 执行层职责：只做守门，不做纠偏
**文件**
- `d:\HLPrj\HL\device\src\slot_transfer_cycle_vtm_widget.cpp`
- 如有必要，补查实际 Robot 请求结构体使用点

**修改内容**
- 所有 `robot_get_from_ll*`、`robot_put_to_pm2`、`robot_exchange_pm2`、`robot_get_from_pm2` 的请求 arm，都必须来自任务规划结果，不能在发送前临时改成另一只手。
- 执行层只回答两件事：该 arm 现在能不能执行；如果不能，是等待、回退还是告警。执行层不再承担帮调度修 arm的责任。

**原因**
- 只要执行层还带纠偏职责，就会重新引入第二条语义通路。

### E. 用日志与验收用例固化 fixed-arm，不给后续回退留空间
**文件**
- `d:\HLPrj\HL\device\src\slot_transfer_cycle_vtm_widget.cpp`
- `d:\HLPrj\HL\device\src\TaskManager.cpp`

**修改内容**
- 新增或统一关键日志关键词：
  - fixed-arm plan built
  - fixed-arm wait/retry
  - rejected runtime arm rewrite
  - exchange plan inconsistent / not issued
- 对照 `d:\HLPrj\HL\流程调度时序.txt` 建立最小验收矩阵：
  1. task0=A 上 PM2，task1=B 预装，task0 完成后 A取/B放。
  2. task1=B 完成后 B取/A放。
  3. 某 arm 已占用但不匹配任务 arm 时，系统等待/回退，不换臂。
  4. PM2 有片但任务对未对齐时，不发交换指令。

**原因**
- fixed-arm 问题本质是语义一致性问题，必须靠计划日志 + 时序验收双重锁定，而不是只靠一次现场跑通。

## Assumptions & Decisions
- 决策 1：`task.arm` 是 wafer 级绑定语义，不允许在任务生命周期内动态改变。
- 决策 2：现场物理手臂状态只参与能否执行的判定，不参与任务 arm 是谁的判定。
- 决策 3：当绑定 arm 不可执行时，允许等待、回退、重调度、告警；不允许偷偷换臂继续跑。
- 决策 4：PM1~PM4 框架保留，但本次根治重点聚焦 PM2 实际命中的交换与 LL 衔接路径。
- 决策 5：为降低编码风险，优先在当前已确认的 UTF-8 源文件上完成治理；若修改 `TaskManager.h` / `UnifiedWaferTask.h`，需先再次确认原始编码与 CRLF，再按原编码保存。

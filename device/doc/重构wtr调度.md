## 优先级设计逻辑

```
优先级从高到低：

  ┌─────────────────────────────────────────────────────────────┐
  │ P1: PUT_TO_LL (放片回LL)                                    │ ← 最高优先级
  │     原因：手臂上有已完成工艺的片，尽快放回LL释放手臂          │    释放资源
  ├─────────────────────────────────────────────────────────────┤
  │ P2: GET_FROM_PM (从PM取片)                                  │ ← 次高
  │     原因：PM工艺已完成，尽快取出释放PM腔体给下一个片使用      │    释放PM
  ├─────────────────────────────────────────────────────────────┤
  │ P3: EXCHANGE_PM (PM交换料)                                  │ ← 中等
  │     原因：一次操作完成取+放，效率最高                        │    效率优化
  ├─────────────────────────────────────────────────────────────┤
  │ P4: PUT_TO_PM (放片到PM)                                    │ ← 较低
  │     原因：送新片进PM开始加工                                 │    启动加工
  ├─────────────────────────────────────────────────────────────┤
  │ P5: GET_FROM_LL (从LL取片)                                  │ ← 最低优先级
  │     原因：取新片只是补充手臂库存，优先处理已在流程中的片      │    补充库存
  └─────────────────────────────────────────────────────────────┘
```

## 线程间通信总结

| 通信方式               | 说明                           |
| ---------------------- | ------------------------------ |
| `requested`            | 业务线程 → Robot线程，发起请求 |
| `done`                 | Robot线程 → 业务线程，通知完成 |
| `success`              | Robot线程 → 业务线程，通知结果 |
| `arm/slot`             | 业务线程 → Robot线程，传递参数 |
| 全部使用 `std::atomic` | 无锁，无互斥量，无阻塞等待     |

核心思想：**所有 wtr 硬件操作集中在一个线程中串行执行**，避免了多线程抢占 wtr 资源的竞争问题。业务线程只负责"提交请求"和"等待结果"，不直接操作硬件。

## 1. 架构问题

| ID | 问题 | 说明 / 证据 |
|----|------|-------------|
| A-01 | **三轨状态并存，无单一真相源** | 同时存在：`TaskManager`（全局任务表）、线程本地缓存（`loadLockAPendingTasks` / `pm2PendingTasks` 等）、遗留布尔标志（`tool_allow_*` / `pm*_allow_*`）、`RobotTransferRequest`（atomic 请求）。与 SchedulerSkills「队列+标志双通道」相比，通道数过多且未统一。 |
| A-02 | **Widget 与 TaskManager 职责边界模糊** | 约 1.2 万行状态机仍在 Widget 内；TaskManager 本应承载任务生命周期，但决策、互锁、PM2 优先级仍在各 `execute*Transfer` 的 `case` 中。 |
| A-03 | **SchedulerV2 命名与实现不一致** | 对外可称 V2，但结构上仍是「老项目多线程 + 步号状态机」叠加 TaskManager/Robot 请求层，未形成独立的 SchedulerService / 决策层 / 执行层。 |
| A-04 | **遗留字段未收敛** | 仍存在 `robot_auto_step`、大量 `tool_allow_*` / `pm*_allow_*`（与老项目同构），与 `robot_step` + `RobotTransferRequest` 并行，增加认知与维护成本。 |
| A-05 | **PM 调度能力不对称** | PM2 有 `case 200` 集中决策、`tryBuildPm2ScheduleSnapshot`、`syncPm2ReturnTaskArm`；PM1/3/4 多为模板式状态机，**全局优先级规则未抽象**，扩展 PM 需复制 PM2 补丁逻辑。 |
| A-06 | **决策出口分散（跑片已确认）** | PM2 规则在 PM2 线程 `case 200`；LL 是否在 `1051` 取片由 `shouldLlWaitForPm2Priority`；能否发 LL 请求由 `isRobotArmOccupiedForLlRequest`；Robot 仅在 `executeTMTransfer case 10` 做优先级扫描——**同一业务规则至少 4 处**。 |
| A-07 | **TaskManager 单例全局耦合** | `TaskManager::getInstance()` 与 UI Widget 强绑定，不利于单测、多配方实例或调度模块独立部署。 |
| A-08 | **workTasks_ 与 tasks_ 双队列语义不清** | `addWorkTask` / `getAllWorksTasks` 与主 `tasks_` 关系文档化不足，易出现「任务在 master 不在 work」类不一致。 |

## 2. 状态机问题

| ID | 问题 | 说明 / 证据 |
|----|------|-------------|
| S-01 | **魔法步号 + 多状态机交织** | `efem_auto_step` / `loadlock*_auto_step` / `pm*_auto_step` / `robot_step` / `vacuum_auto_step` / `update_auto_step` 并存，语义靠区间记忆，不符合可重构的表驱动模型。 |
| S-02 | **UnifiedWaferTask 单槽位 (type, status) 表达多阶段生命周期** | 一片 wafer 需经 EFEM→LL→PM→LL→EFEM；`updateTaskStatus` **覆盖** `taskType`+`status`，历史阶段仅依赖 `taskTypeStatusMap_` 索引，无显式「阶段栈」或并行子状态。`ROBOT_PROCESS` / `RobotStatus` 枚举在头文件中存在但主路径几乎未用。 |
| S-03 | **task.arm 语义混用（跑片已确认）** | `arm` 同时表示配方规划手与运行时 `updateTaskArm` 修改后的实际手；LL/PM2 回片、交换、立即补取均依赖 `task.arm`，无法区分计划 vs 执行。 |
| S-04 | **状态推进非事务化（跑片已确认）** | 交换片→回片登记 `LOADLOCK_RETURN`→LL 放回→立即补取跨线程、跨 case 完成；中间窗口导致 LL 按错手等待（已用 `syncPm2ReturnTaskArm` 等补丁缓解）。 |
| S-05 | **PM2 case 200 规则堆叠（跑片已确认）** | 单 case 内同时判断 PM 占片、双臂持片、pending/completed/return、放/取/交换；大量 `10→200→10` 空转与兜底分支，说明规则未形式化。 |
| S-06 | **LLA/LLB 双份状态机（跑片已确认）** | `950/1051/1055/2060/2066` 等逻辑 LLA/LLB 各写一套，修复易漏边。 |
| S-07 | **立即补取（LLImmediateRepick）与 Robot expedited 双结构** | LL 侧 `llaImmediateRepick` 记计划，Robot 侧 `expedited` 提优先级；两段状态需手动对齐，易漂移。 |
| S-08 | **Robot 请求生命周期不完整** | `requested/done/success` 无统一超时、无「in-flight 所有者」；`1015` 类等待步靠轮询，异常时 `done=false && requested=false` 需特判重发。 |
| S-09 | **updateTaskStatus 与本地缓存不同步** | `UpdateLLASubTransferDatas()` 刷新 `loadLockAPendingTasks` 后，同一线程仍可能读写 `loadLockAPendingTasks.at(0).arm` 与 TaskManager 不一致（如 `updateTaskArm` 只改一端）。 |

## 3. 线程安全问题

| ID | 问题 | 说明 / 证据 |
|----|------|-------------|
| T-01 | **TaskManager::clearTasks 无锁** | `clearTasks()` 直接 `tasks_.clear()`，与并发 `updateTaskStatus` / `get*Tasks` 竞态。 |
| T-02 | **TaskManager::getAllTasks / getAllWorksTasks 返回内部 vector 引用且无锁** | 锁释放后调用方遍历引用，与写线程数据竞争（潜在 UB）。 |
| T-03 | **getAllTasksSize 无锁** | 多线程用 size 做分支时可能读到撕裂值或与内容不一致。 |
| T-04 | **getByIDFindTask / getRobotTaskInfo 无锁** | 遍历 `tasks_` 无 mutex；未找到时 `getByIDFindTask` 无 return（未定义行为）。 |
| T-05 | **getTaskType 返回 map 内引用，锁外使用危险** | 锁在函数返回后释放，返回 `it->second.first` 引用可被后续更新破坏。 |
| T-06 | **setLoadingInPlaceFlag 只改局部副本** | `auto task = getByIDFindTask(...); task.isLoadingInPlace = flag` 未写回 `tasks_`（若被调用则标志无效）。 |
| T-07 | **大量 bool 标志非 atomic** | `tool_allow_*`、`pm*_allow_*` 等多线程读写，依赖 Sleep 轮询可见性，无 memory order 保证。 |
| T-08 | **线程本地任务 vector 与 TaskManager 双写** | 如 `loadLockAPendingTasks.at(0).arm = targetArm` 与 `taskManager.updateTaskArm` 需成对出现，遗漏即不一致。 |
| T-09 | **10 路 detach 线程 + 粗粒度 running** | `startProcessingThreads` detach；暂停靠 `mtx`+`cv`+`running`，Reset/Abort 与运行中线程交错时序复杂。 |
| T-10 | **pm*_auto_step 为 atomic，loadlock*_auto_step 非 atomic** | 同类步号变量线程安全策略不统一。 |
| T-11 | **CheckableMutex / feeding_mutex 使用面不清晰** | 与 TaskManager 锁、Robot 请求 atomic 叠加，锁顺序未文档化，增加死锁排查难度。 |

## 5. 潜在死锁问题

| ID | 问题 | 说明 / 证据 |
|----|------|-------------|
| D-01 | **逻辑死锁：多向等待无超时（跑片相关）** | PM2 `case 200` 与 LL `1051` 互等：PM 等 pending/return 对齐，LL 等 `shouldLlWaitForPm2Priority`，Robot 未消费请求时各方 `Sleep` 轮询，表现为长时间 ``10↔200`` 或 ``950↔1051`` 空转（非 OS 死锁，但是**调度僵死**）。 |
| D-02 | **标志粘滞导致永久等待** | 若 `requested=true` 后 Robot 线程异常未清 `done`，业务线程在 `1055/1015` 无限等待；无 watchdog。 |
| D-03 | **pm2_exchange_in_flight 与 robot_exchange_pm2.requested 双标志** | PM2 等待 Robot 消费交换；Robot 失败若未清 `pm2_exchange_in_flight`，PM2 在 `7084` 附近持续 break 等待。 |
| D-04 | **Vacuum 与 WTR 互锁依赖轮询** | Vacuum 等 `getWtrOriginSafeSignal` && `!isBusy`；WTR 可能等真空；若信号异常，双方长期等待（SchedulerSkills §5.3 类风险）。 |
| D-05 | **上料互锁 isLoadingInterlock 与双 LL 交叉** | 一侧下料未完成阻塞另一侧上料，若标志/任务状态未清，可形成**业务层互锁**（LP/LL/EFEM 三角等待）。 |
| D-06 | **双向 cv 使用相对安全但 Abort 路径复杂** | `pauseAllThreads` notify_all 后各线程退出 wait；若某线程在 `cmd->wait()` 阻塞，与 `running=false` 不同步，表现为「停不住」而非经典死锁。 |
| D-07 | **TaskManager 锁与设备命令 wait 嵌套风险** | 规范要求不在持锁下 `cmd->wait()`；若后续维护在 `updateTaskStatus` 回调链中调硬件，可能拉长锁持有时间，间接阻塞其他线程查询。 |
| D-08 | **队首任务饥饿** | Robot 优先级固定，若高优先级请求源持续置位，低优先级 `GET_FROM_LL` 长期得不到服务（逻辑饥饿）。 |

## 6. 与 SchedulerSkills 的对照差距

| SchedulerSkills 约束 | SchedulerV2 现状 |
|----------------------|------------------|
| WTR 单线程执行 | ✅ `executeTMTransfer` 集中下发（符合） |

## 8. 重构建议优先级（仅规划）

1. **P0**：修 TaskManager 线程安全（clear/getAll/getByID/getTaskType）；统一任务读写入口。  
2. **P0**：引入**调度快照**（硬件 + TaskManager 一次采集），单一 `decideNextAction()`。  
3. **P1**：拆分 `plannedArm` / `runtimeArm`；状态推进事务化（动作完成回调单点更新）。  
4. **P1**：LLA/LLB 共用 LL 调度器；PM 决策从 case 200 抽到 `PmInteractionPolicy`。  
5. **P2**：Robot 请求统一生命周期（超时、重试、owner）；淘汰冗余 `tool_allow_*` 或映射到 Gate 对象。
